# Nerospace privacy & telemetry disclosure

Nerospace includes optional, anonymous error reporting so that crashes and bugs caused by
the mod can be found and fixed. This page is the full disclosure required by CurseForge's
moderation rules for mods that use an external analytics/error service, and it documents
how the system is designed to comply with the GDPR (EU) and POPIA (South Africa).

## What is collected

When an error **caused by Nerospace code** occurs, a report is sent to
[Sentry](https://sentry.io) (hosted in the **EU** — `ingest.de.sentry.io`) containing:

- the stack trace of the error and the error message (file paths containing your
  operating-system account name are masked before sending),
- the Nerospace version, the loader (NeoForge / Forge / Fabric) and its version, the
  Minecraft version, and whether the game is a client or dedicated server,
- the operating system name/version and Java version,
- the mod's own configuration multipliers (e.g. energy/oxygen/fuel/machine-speed settings),
  so a crash that only happens at non-default balance settings can be reproduced,
- the list of **other loaded mods** (their public ids and versions only), so a crash caused
  by a mod conflict can be identified,
- a short trail of recent in-game mod actions ("breadcrumbs") — for example "launched a
  rocket", "travelled to a dimension", "founded a station" — recorded as generic action
  names plus dimension/slot ids only (never coordinates, inventory, or world data),
- a random event ID generated by Sentry for the individual report.

To measure overall stability and performance, the mod also sends:

- **Release-health sessions:** a count that a play session started and whether it ended
  cleanly or crashed, tagged only with the mod / loader / Minecraft version. Each session
  uses a random identifier that exists only for that single play session and is **not**
  linked to any other session, to your account, or to your machine.
- **Performance samples:** the timing of a small, randomly-sampled fraction of internal mod
  operations (such as a rocket launch or a pipe-network rebuild) — duration and operation
  name only, with no personal data.

## What is NOT collected

- **No IP address** is stored (`sendDefaultPii` is disabled; Sentry discards the sender IP).
- **No username, player UUID, or machine/host name.**
- **No world data, chat, coordinates, inventories, or gameplay analytics** of any kind.
- **No advertising or tracking identifiers** — there is no persistent ID linking reports
  from the same player or machine.
- Errors caused by other mods or by Minecraft itself are filtered out and never sent.

Reports are also de-duplicated and hard-capped at 10 events per game session.

## How to opt out

Set the following in `config/nerospace.properties` (created on first launch):

```properties
telemetryEnabled=false
```

This single switch covers **everything** the mod sends — error reports, release-health
sessions, performance samples, breadcrumbs, and the loaded-mod list alike.

The change takes effect immediately on config reload (or next launch) and no data is sent
while disabled. Server owners can opt out for the whole server the same way. Nothing is
ever sent before the config — and therefore your choice — has been loaded.

## Legal basis and your rights (GDPR / POPIA)

Error reports are processed on the basis of **legitimate interest** (GDPR Art. 6(1)(f);
POPIA s11(1)(f)): keeping the mod functional and crash-free, using the minimum data
necessary. The reports are engineered to contain no directly identifying personal
information, are stored in the EU, and are retained only as long as Sentry's standard
retention period (90 days) before automatic deletion.

If you believe a report contains personal data, or wish to exercise any data-subject
right (access, deletion, objection), contact:
**[info@neroland.co.za](mailto:info@neroland.co.za)**. Include the
approximate date/time of the crash so the matching event can be located and removed.

Sentry acts as a data processor; see the
[Sentry privacy policy](https://sentry.io/privacy/) and
[data processing addendum](https://sentry.io/legal/dpa/).

## Changes to this policy

If a future version of Nerospace ever collects additional data, this page will be updated
**before** that version is released, the change will be called out in the changelog, and the
same opt-out (`telemetryEnabled = false`) will continue to cover everything the mod sends.
The policy shipped alongside a given mod version describes exactly what that version does.

## Developer note: verifying telemetry locally

No env file or credentials are needed — the DSN is embedded. Launch a dev client with the
self-test flag and one synthetic event is sent on config load (environment matches the
version channel, with tag `runtime=dev`):

```text
./gradlew runClient -PtelemetryTest=true
```

or add `-Dnerospace.telemetry.test=true` to the IDE run config's VM args. Check the log for
"Telemetry self-test" and the event in the Sentry dashboard. Setting
`telemetryEnabled=false` in `run/config/nerospace.properties` and reloading must stop
events — that's the opt-out test.

For verifying a **release jar** end-to-end, the mod also registers a hidden diagnostic block,
`nerospace:sentry_test`. It does not appear in the creative menu and cannot be crafted — the
only way to obtain it is the command `/give @s nerospace:sentry_test`. Placing it sends a
single synthetic, clearly-labelled test event ("safe to ignore") through the exact same
pipeline — and therefore the same scrubbing, opt-out gate, and session cap — as a real error
report. If telemetry is disabled it sends nothing and says so in chat. Regular gameplay can
never trigger it.
